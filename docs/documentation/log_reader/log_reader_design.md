# Log Reader — Implementation (v3)

## Objective

Implement a log reader module for the airline booking backend pipeline.

The goal of this phase is to:

- Read synthetic log files generated by `log_generator.py`
- Return raw log lines from all files for a given service
- Support reading all files across all services
- Handle missing or empty log directories gracefully
- Prepare data for the parsing stage

---

# System Context

The log reader is the first stage of the ingestion layer.

It sits between the raw log files on disk and the parser. Its only
responsibility is to access the file system and deliver raw content.
It does not know anything about the log format or structure — that is
the parser's responsibility.

```
log_generator.py → data/raw/<service>/ → log_reader.py → log_parser.py
```

# Public Interface

## `load_service_logs(service)`

Reads all log files for a given service and yields their lines.

**Parameters:**
- `service` (str) — name of the service to read logs from (e.g. `"booking"`)

**Returns:**
- `Iterator[str]` — yields one string per log line across all files

**Raises:**
- `ValueError` — if the service directory does not exist
- `FileNotFoundError` — if the service directory exists but contains no log files

---

## `load_all_logs(services)`

Reads all log files across all services and yields their lines.

**Parameters:**
- `services` (list[str]) — list of service names loaded from config

**Returns:**
- `Iterator[str]` — yields one string per log line across all services and files

---

# Private Functions

## `_load_all_path_names(services)`

Builds the full path for each service directory.

**Parameters:**
- `services` (list[str]) — list of service names

**Returns:**
- `list[Path]` — list of paths to service directories

**Implementation Details:**
- Resolves the project root using `__file__`
- Constructs `data/raw/<service>` for each service name

---

## `_load_all_files(paths)`

Collects and sorts all log files across the given paths.

**Parameters:**
- `paths` (list[Path]) — list of service directory paths

**Returns:**
- `list[Path]` — sorted list of all log files found

**Implementation Details:**
- Iterates over each path and collects files using `is_file()`
- Sorts all files alphabetically for deterministic ordering

# Implementation Details

## Path Resolution

Uses `pathlib` for OS-independent path resolution. Resolves the project
root dynamically using `__file__` to ensure portability across
environments including Docker.

```
src/ingestion/log_reader.py
    → parents[2] resolves to project root
    → project_root / data / raw / <service>
```

## File Selection

Collects all files in the service directory, sorted alphabetically
for deterministic ordering. Only files are included — subdirectories
are excluded via `is_file()`. The sort ensures the oldest files (by
naming convention) are processed first, making the pipeline reproducible
across runs and operating systems.

## Return Format

Both public functions use `yield` to deliver one log line at a time
as a generator. Lines are delivered on demand as the caller iterates,
avoiding loading all files into memory at once. The parser receives
the generator and processes each line independently. Because Python's
for loop works with any iterable, the parser does not need to know
whether it receives a list or a generator.

## Error Handling

### `load_service_logs`
Validation happens before any file access — fail fast principle.

```
1. Check if service directory exists → ValueError if not
2. Check if directory contains files → FileNotFoundError if empty
3. Read and yield lines from all files
```

Error messages include the service name for clarity:
- `"The service {service} does not exist"`
- `"There are no logs for {service}"`

### `load_all_logs`
No explicit validation. The service list comes from `config.yaml`
and the directories are created by the generator from that same
config. If a directory is missing, `pathlib` raises a clear error
on iteration. Downstream validation in `_verify_columns()` catches
the case where no data reaches the DataFrame.

# Design Decisions

## Two public functions instead of one with dual behavior
`load_service_logs` handles a single service with input validation
because the service name comes from user input via CLI.
`load_all_logs` handles all services without validation because the
service list comes from config. Keeping them separate avoids a
function that changes behavior based on parameters.

## Helper functions for multi-service reading
`load_all_logs` delegates path building and file collection to two
private helpers. Each function has a single responsibility and a
single level of iteration. This avoids three levels of nesting
(services → files → lines) in one function.

## No validation in load_all_logs
Services come from config.yaml, which is the same config the
generator uses to create the directories. Validating that the
directories exist would be redundant. If they are missing, pathlib
produces a clear error. If no data reaches the pipeline, the
existing `_verify_columns()` catches empty input.

## All files per service
The reader iterates over all files in each service directory. Files
are processed in alphabetical order, which matches the generator's
timestamp-based naming convention. This ensures analysis covers the
full dataset, not just a single file.

## Raw strings, no processing
The reader returns raw strings. No parsing, no transformation. This
maintains a clean separation of responsibilities between ingestion
and processing layers.

## Pathlib over os.path
`pathlib` provides a cleaner and more readable API for path operations
and is the modern Python standard for file system handling.
