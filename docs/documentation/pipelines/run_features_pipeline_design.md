# Run Features Pipeline — Design (v1)

## Objective

Orchestrate the feature engineering workflow for the log analysis
pipeline. Receives a validated DataFrame and configuration, delegates
feature generation to the feature engineering module, and persists
the resulting dataset to disk as CSV.

---

## System Context

The features pipeline sits between the analysis layer and the file
system. It connects the validated DataFrame with the feature
engineering module and handles output persistence.

```
main.py → run_features_pipeline.py → feature_engineering.py → output/datasets/features.csv
```

## Function: run_features_pipeline()

### Parameters

- `logs_dataframe` — pandas DataFrame with validated log data
- `raw_data` — full config dict loaded by `main.py`

### Returns

- None — output is saved to disk

### Implementation Details

- Calls `_make_output_directory()` to ensure the output path exists
- Constructs the output filename as `features.csv`
- Extracts `feature_thresholds` and `service` from the config dict
  and passes them to `orchestrate_features()`
- Receives a new DataFrame containing context columns and derived
  features
- Persists the feature DataFrame to CSV using `.to_csv()` with
  `index=False`

### Pipeline Flow

```
1. _make_output_directory()                          → Path
2. orchestrate_features(df, thresholds, services)    → feature DataFrame
3. features_dataframe.to_csv(path, index=False)      → output/datasets/features.csv
```

### Design Decisions

- **Config extraction happens in the pipeline, not in the feature
  module.** The pipeline knows the config structure and extracts
  `feature_thresholds` and `service` before passing them to
  `orchestrate_features()`. The feature module receives only what
  it needs — same decoupling pattern used across the project.

- **`index=False` on CSV persistence.** The pandas index is an
  autogenerated sequence with no relationship to the log data.
  Saving it would create an extra column (`Unnamed: 0`) when the
  CSV is read back, requiring cleanup. Excluding it keeps the
  dataset clean for downstream consumption.

- **Separate pipeline from `run_pipeline.py`.** The data pipeline
  produces a validated DataFrame — that is its single
  responsibility. Feature engineering is a distinct transformation
  with its own output artifact (CSV dataset vs DataFrame). Keeping
  them separate follows the same pattern as the reporting pipeline.

- **Called by `main.py`, not chained to `run_pipeline`.** Pipelines
  do not call each other. `main.py` orchestrates the sequence:
  data pipeline → reporting pipeline → features pipeline. This
  keeps each pipeline independent and testable.

## Function: _make_output_directory()

### Parameters

- None

### Returns

- `Path` — path to the output directory for datasets

### Implementation Details

- Uses `pathlib` for OS-independent path resolution
- Resolves the project root using `__file__`
- Creates `output/datasets/` if it does not exist using
  `mkdir(parents=True, exist_ok=True)`

### Design Decisions

- **Separate from `report_pipeline`'s `_make_output_directory()`.**
  Both pipelines create output directories, but for different
  purposes (`plots/` vs `datasets/`). Sharing a single function
  would require parameterization that adds complexity without
  clear benefit at this stage. If a third pipeline introduces
  another output directory, extracting a shared utility can be
  reconsidered.
