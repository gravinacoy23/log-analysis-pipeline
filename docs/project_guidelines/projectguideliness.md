# Project Guidelines – Log Analysis Pipeline

## Project Purpose

This project is a 8-10 month learning journey combining:

- Python
- Data Analysis
- Linux
- Git and Documentation
- Cloud (AWS)
- Machine Learning fundamentals

The goal is to simulate a real-world data pipeline environment while
building practical skills usable in industry.

The project simulates an airline booking backend as a vehicle for learning.
The domain is not the point — the engineering fundamentals are.

---

## Core Rules

### 1. Always Use Git

Every work session must include:

- Pulling the latest changes before starting
- Making small, frequent commits
- Writing meaningful commit messages
- Pushing changes to GitHub before ending the session

Example commit messages:
- "Add log reader with file-not-found handling"
- "Refactor load_config to validate required keys"
- "Add response_time correlation to CPU in generator"

---

### 2. Documentation Is Mandatory

Every session must include documentation updates stored in `docs/`.

Documentation can include:
- What was done
- What was learned
- Problems encountered and how they were solved
- Design decisions and why they were made

---

### 3. Prefer learning in depth than learning fast.

The goal is learning well all the concepts required to advance in carreer.

Prefer:
- well written scripts
- Readable code
- Clear structure

Avoid:
- Over-engineering
- Large frameworks before understanding fundamentals
- Unnecessary abstractions

---

### 4. Work With Realistic Data

Logs are synthetic but designed to be realistic and ML-ready.

Logs will:
- Be generated by `log_generator.py`
- Include realistic correlations between metrics
- Be stored as structured text files
- Be parsed using Python and analyzed using pandas

### 6. Fail Fast

Validate inputs early. If a required config value is missing or empty,
raise a descriptive error immediately rather than letting the pipeline
fail silently later.

## Session Workflow

Each session should follow this structure:

1. Pull repository
2. Give the assistant a quick summary of where you left off
3. Share relevant MD files at the start of a new chat
4. Work on a small, focused task
5. Test the code
6. Document the work
7. Commit changes with a meaningful message
8. Push to GitHub

---

## Learning Philosophy

- **Depth over speed.** Taking longer to understand something well is
  always better than moving fast and understanding nothing.
- **No vibe coding.** Every line of code should be understood before
  it is written.
- **Fail and learn.** Errors are part of the process. Try before asking.
- **Gradual independence.** As the project progresses, attempt full
  sessions without assistant help and only review at the end.

## Rules for the Assistant (Scrum Master)

The assistant should:

- Guide without giving solutions directly
- Ask questions that lead the learner to the answer
- Provide small, focused daily tasks
- Connect every task to the main project
- Gradually increase difficulty
- Encourage documentation and commits every session
- Regarding documentation, the assistant will generate all design docs, Juan will write doc strings.
- Everytime a future improvement is identified in the design of a module, in the documentation, the assistant should also remind to add it to tech debt.
- Flag when the learner is over-engineering or jumping ahead
- Remind the learner to try things before asking

The assistant should NOT:
- NEVER recommend something that's easier to implement but goes against python and programming best practices, even if it will take long to implement
- Skip fundamentals
- Introduce unnecessary complexity
- Jump too far ahead in the roadmap
- Give code directly unless the learner is stuck after trying

### Context Management
- At the start of each new chat, the learner should share a brief
  summary of where they left off
- Relevant MD files should be shared at the start of each new chat
- Consider starting a new chat per major module to keep context clean
